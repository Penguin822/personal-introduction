全端電影購票與劃位系統

這個專案我使用了比較經典的網頁開發技術組合（HTML、CSS 和 PHP）來打造完整的電影購票系統。雖然作法比較傳統，但非常考驗對前後端資料傳遞與資料庫狀態管理的掌握度。

1. 資料庫建模與綱要設計

在寫程式前，我先針對電影場次、座位狀態、會員訂單等關聯性繪製了 ERD，並建立完整的 Schema。這確保了在遇到多位使用者同時劃位時，資料庫結構能支撐複雜的查詢與寫入。

▲ 資料庫關聯圖 (ERD) 與綱要設計 (Schema)
2. 全端網頁開發與 UI 實作

前端部分：我運用 HTML 搭建語意化架構，並親自用 CSS 設計了直覺的電影瀏覽介面，特別是「視覺化的座位選取」功能，讓使用者體驗更順暢。
後端部分：導入 PHP 處理核心商業邏輯，包含會員登入驗證、動態場次查詢、購物車狀態管理，以及最後的訂單寫入 (CRUD) 操作。

學生憂鬱症成因分析與預測模型

這是一項結合 Kaggle 競賽數據的資料探勘實務專案。全程使用 Python 開發，並大量運用 Scikit-learn 等機器學習套件。

▲ 採用 Kaggle 競賽數據集進行實作
1. 預處理與特徵選取：找出真正的訊號
「在這個專案中，我認為最關鍵的不是直接套用模型，而是前期的特徵工程。在完成資料預處理後，我運用了 Information Gain 和 Gain Ratio 來量化每個屬性對結果的影響力。這就像是在龐雜的數據中，精準篩選出真正的『訊號』，排除掉不相關的雜訊。」
▲ Information Gain 與 Gain Ratio 演算法特徵評估結果
2. 多模型實作與效能對標

我建構了 5 種機器學習模型：貝氏分類器 (Naive Bayes)、KNN、SVM、隨機森林 (Random Forest) 以及 XGBoost。在評估模型表現時，我設計了一套非常嚴謹的機制：

  • 混淆矩陣分析：我絕對不會只看「準確率 (Accuracy)」。我知道預測錯誤的代價是不同的 —— 就像在 IMC 的交易策略中，「錯過一次賺錢機會 (偽陰性)」跟「執行了一次賠錢交易 (偽陽性)」，兩者對公司的影響完全不同。混淆矩陣讓我能清楚掌握模型到底犯了哪種錯。
  • ROC 曲線比較：驗證模型是否真的具備區分訊號的能力，而不是剛好在某個閾值下賽中。
  • 交叉驗證 (Cross-Validation)：最後一步，所有模型我都經過了交叉驗證。這確保了最終選出的 XGBoost 不是處於過擬合狀態，而是具備極高的穩定性。

我相信這種對於「防範過擬合」與「嚴謹驗證」的堅持，正是在開發高頻交易模型時最不可或缺的態度。

AWS 雲端自動化設備維護與監控系統

我的第四個專案是利用 AWS 打造的設備維護管理系統。這個專案展現了我從「實體基礎設施」到「無伺服器 (Serverless) 架構」的整合能力。

▲ AWS 系統整體架構圖
1. 雲端基礎設施部署與資安控管

首先，我配置了一台 Ubuntu 的 EC2 作為網頁伺服器，並開通 RDS 作為底層資料庫。為了確保架構安全，我在本機端設定資料庫 Schema 時，嚴格控管了 RDS 的 Security Group，只允許特定 IP 進行連線操作，確保資料不外洩。

▲ EC2 伺服器部署與 RDS 安全群組 (Security Group) 設定
2. 無伺服器自動化排程監控 (Serverless)

基礎架構建置好後，我將重點放在「自動化監控」上。我導入了 AWS EventBridge 取代傳統的主機排程器。 當時間觸發時，系統會自動啟動 Lambda 函數去 RDS 掃描設備的維護期限，一旦發現即將到期，就會透過 SNS 主動發送 Email/簡訊告警，達成完全不需要人工介入的自動化維運。

▲ 運用 EventBridge 觸發 Lambda 進行 Serverless 自動化排程
